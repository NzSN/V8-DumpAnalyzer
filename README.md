# V8-DumpAnalyzer

Post-mortem V8 JavaScript stack trace extractor from Windows minidumps.

Reads `.dmp` files (generated by Crashpad when Electron/Chrome crashes) and extracts:
- Native stack trace with crash exception info
- JavaScript function names from V8 interpreted frames
- V8 heap and Isolate metadata

## Features

- **Self-contained** single-file implementation. No external dependencies.
- **Heuristic scan** finds `c_entry_fp` from Isolate memory — no V8 source modifications needed.
- **Full memory dumps** (all committed memory included).
- Reads `v8-oom-stack` annotation for OOM crash JS traces.
- **Standalone GN build** with DEPS for V8, or **simpe MSVC build** via `scripts\build.bat`.

## Build

### Quick build (no V8 dependencies):

```
scripts\build.bat
```

Output: `out\dump_analyzer.exe`

### With V8 headers (GN + DEPS):

First pull V8 and build tools:
```
gclient sync
```

Then build:
```
gn gen out\Release
ninja -C out\Release dump_analyzer
```

## Usage

```
dump_analyzer crash.dmp
```

## Design

### Frame Layout

V8 stores interpreted JS function frames on the C stack:

```
fp + 8:  return address
fp:      saved RBP
fp - 8:  context (compressed Tagged pointer)
fp - 16: JSFunction (compressed) → shared() → SharedFunctionInfo → Name()
fp - 24: argument count
fp - 32: BytecodeArray (compressed) [unoptimized only]
```

### Finding c_entry_fp

The `c_entry_fp` (C entry frame pointer) is the boundary between C++ and JS frames.
It's stored in the V8 `Isolate.thread_local_top_.c_entry_fp_` field (private).
The tool scans the Isolate's first 4096 bytes for values in the crash thread's
stack range, then verifies by checking the return address at `val+8` is in V8 code.

### File format

The minidump is parsed directly (no Crashpad dependency required).
Structures parsed: MemoryList, ThreadList, Exception, CrashpadInfo annotations.
